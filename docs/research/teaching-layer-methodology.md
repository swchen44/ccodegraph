# 教學層方法論:怎麼寫一份讓 agent 多拿分的 SKILL(2026-07-13)

> **TL;DR**:六輪 benchmark 的受控數據顯示,**工具的天花板由教學層
> 決定,不由引擎決定**——同一套紀律教學,給 ccodegraph 帶來 **+4/66**
> (v4:58→62),給 clangd LSP 帶來 **+1/66**(v6.5:60→61),而單句
> 「使用時機提示」帶來 **+0**(9-run 探針)。本文把「什麼教學有效、
> 什麼無效、怎麼驗證」寫成可複用的方法論。所有數字可追溯:
> `llm-ab-v4-token-efficiency.md`、`llm-ab-v6-lsp.md`。

## 0. 為什麼教學層是獨立的工程物件

四個獨立證據指向同一結論:

1. **codegraph(v3/v5)**:功能完整的圖工具,無教學層——agent 跑了
   14 次 `--help`、單一 verb 回傳 192KB,分數與 grep 打平。
2. **cbm(v3)**:同上,57/66,輸給有 SKILL 的 ccodegraph(58)。
3. **clangd(v6)**:業界正統,九個語意操作在場——36% 的 runs 一次
   都沒用,殺手鐧 `incomingCalls` 66 runs 只被用 4 次;60/66 = grep。
4. **SWE-agent ACI(NeurIPS 2024)**:介面呈現方式比工具本身更影響
   agent 表現——上面三條是它的自然實驗版。

推論:**你的工具沒有教學層,它對 agent 就只值 grep 的分數**——
不管引擎多強。

## 1. 有效教學的四種類型(附受控證據)

### 1.1 領域陷阱課(最值錢,可跨工具移植)

把「這個領域會怎麼騙你」寫成一句可執行的規則:

- **雙閘控**:「config 相依的程式碼有兩層閘:檔內 `#ifdef` **和**
  build 系統(`Makefile OBJS +=`/Kconfig)——兩層都要查。」
  證據:v4 教給 ccodegraph 後 WRQ-013 拿 3;v6.5 移植給 clangd 的
  SKILL,同題 2→3。**同一堂課,兩個引擎都吃**。
- **雙拼寫**:「`#include "x.h"` 與 `#include "dir/x.h"` 兩種寫法都要
  數;`#ifdef X` 與 `defined(X)` 亦然。」證據:v1 的 T5(grep 臂靜默
  漏 58 檔)→ 教學化後 v6.5 WRQ-015 2→3。

### 1.2 覆核紀律(通用,與工具無關)

「計數/枚舉題,用兩種獨立方法互核,數字一致才回答;不一致以較大
集合為搜尋空間。」證據:hint-probe 中**升分的 runs 一次工具都沒用**
——起效的就是這半句(WRQ-016 負對照 2→3 證死);它同時是 clangd
findReferences 靜默不完整(實錘:回 4/數百無警告)與 cscope 幻影的
唯一解毒劑。

### 1.3 操作×題型對應表(cheatsheet)

「誰呼叫 X → `prepareCallHierarchy`+`incomingCalls`;定位 →
`workspaceSymbol` 先行」。證據:v6.5 讓 call-hierarchy 使用從 5 次
升到 26 次、漫射式 workspaceSymbol 從 32 降到 11——**改變使用組成
有效**。但注意:組成變準只帶來部分分數;別高估這類教學的權重。

### 1.4 信任校準(工具作者最該寫、最少人寫)

每個輸出「怎麼錯」:confidence 階梯、`ambiguous` 標籤語意、
「findReferences 可能靜默不完整」、「單 config 下哪些檔案不可見,
如何用 grep 補並在答案中揭露」。證據:v6 smoke 中 LSP 臂能正確補
盲區並揭露,正是 prompt 提及單 config 的效果;ccodegraph SKILL 的
風險判讀章是其 62→63 分穩定性的基礎。

## 2. 無效或低效的教學(同樣有受控證據)

1. **單句使用時機提示**:「請優先用 X」「這類問題用 X」——9-run
   探針,使用率幾乎不動(9 個 run 只 2 個碰工具)。**agent 的工具
   先驗是訓練分佈層面的事**(Claude Code 本身就是 grep-first 調教
   的),一句話撼不動。
2. **儀式性步驟**:「Step 0 必跑 schema/status」——v3 實測全是浪費
   (`--help`×7 + `schema`×12 + `status`×6),v4 移除後省 32% 成本、
   分數反升。教學要教判斷,不是教流程。
3. **期待教學修「耐力」**:長清單列不完 → 教了分段策略後行為改善
   有限;真正該修的是**評分視窗**(v6.1:看起來像耐力問題的,實際
   是評分者只看到前 4000 字元)。先驗證是誰的問題再教。

## 3. 工程規範(從 v4/v6.5 的迭代流程提煉)

1. **從 transcript 證據出發**,不從想像出發:每一條教學對應一個
   實測失敗模式(v6.5 的 SKILL 五章,每章都有 66-run 遙測的出處)。
2. **反污染紅線**:教學只放可泛化規則,零 benchmark 題目符號——
   「教規律」與「漏答案」的分界線是:換一個 repo 這句話還成立嗎?
3. **觸發率是第一驗收**:skill 描述要讓 agent 想得起來用(v4 踩過
   5 個 agent 只 1 個載入;修 description 至「Invoke FIRST」+ 觸發
   詞後 100%)。量測法:transcript 掃 `Skill` tool_use。
4. **使用遙測是第二驗收**:工具呼叫次數、操作組成、零使用 run 比例
   ——「教了」不等於「用了」,「用了」不等於「用對」。
5. **N=3 才下結論**:v6.5 的 WRQ-019 smoke(N=1)給 3,全量中位 2。
6. **迭代要留版本與動機記錄**:v1→v2 改了什麼、為什麼、事後動機
   是否被推翻(v6.5 的分段教學動機就是評分假象,如實記錄)。
7. **大小紀律**:6-7KB 上限。教學層自己也吃 token;v4 曾把 13KB 減
   到 6KB,分數不降成本降。

## 4. 一頁式範本

```
---
name: <tool>-nav
description: Invoke FIRST, before <任務類型> —— <一句話價值> + 觸發詞
---
# <工具> playbook
一句話心智模型(它看得見什麼、看不見什麼)
## 操作×題型對應表(cheatsheet)
## 前置工作流(定位/熱身/健檢,各一行)
## 信任校準(每種輸出怎麼錯 + 解毒劑)   ← 最重要的一章
## 領域陷阱(雙閘控/雙拼寫/…,每條一句可執行規則)
## 答題紀律(枚舉列完、總數自核、兩法互核、方法揭露)
```

## 5. 誠實的邊界

- 教學層的可轉移部分(§1.1/1.2)**任何人都能抄**——它是公共財,
  不是護城河;護城河只能是引擎原語(教不出「who-includes」查詢)。
- 分數增益取決於「引擎原語 × 題型」的交集大小:交集大(ccodegraph
  × 枚舉題)+4,交集小(clangd × 同題庫)+1。寫教學層之前先估交集。
- 本文結論來自 C 語言導航問答;編輯迴路(diagnostics)的教學層
  方法論未經受控驗證。
