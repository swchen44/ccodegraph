# Benchmark 方法論:六輪 LLM 工具評測換來的規範(2026-07-13)

> **TL;DR**:評測 LLM+工具的正確性,最難的不是跑實驗,是**讓每一層
> 證據都可信**。六輪(v1-v6.5,504 個評分格、42+ 份 GT)踩出四類
> 「評分管線靜默說謊」的方式與解法、GT 建構的硬規範、以及「oracle
> 也要驗」的差分測試法。全部案例可追溯到 repo 內報告與原始 JSON。

## 1. 評分管線的四種靜默謊言(每種都真實發生過)

| # | 謊言 | 實例 | 解法 |
|---|---|---|---|
| 1 | **GT 本身錯** | WRQ-002:GT 說無同名衝突,deps/hiredis 裡就有一個 static 同名(cbm 的答案暴露它);WRQ-013 的 GT 修正案自己又錯一次 | GT 必須雙方法交叉驗證;**被評者的異常答案是 GT 的免費紅隊**——分數反常先查 GT 再查答案 |
| 2 | **評分者信錯的 GT** | 同上游污染:GT 錯 → 評分者忠實地把對的答案打錯 | 分數與理由的程式化矛盾掃描 + 反常格人工複核,列為固定流程 |
| 3 | **評分者看到的 GT 視圖殘缺** | v5/LKQ-006:GT 檔尾的 SUBTREE ADDENDUM 被 7000 字元截斷切掉 → 全臂統一誤判 | GT 進 prompt 前把否決性段落前置;任何截斷都要驗證「切掉了什麼」 |
| 4 | **評分者看到的答案視圖殘缺** | v6.1:答案視窗 4000 字元,96 項清單被切 → 「明明列完」被判「中途斷掉」,三臂 27 個格子受影響 | 視窗 ≥ 全體答案最大長度,並把「掃描超窗答案」列為收卷前固定稽核 |

**元規則:凡是「評分者看到的東西」——GT、答案、rubric——都是工程
產物,工程產物就會壞;每輪收卷前跑一次視圖完整性稽核。**

## 2. GT 建構規範(v2-v5 演化定形)

1. **兩種獨立方法交叉驗證**每個計數/枚舉(grep 兩種模式、或
   grep×工具、或人工讀碼);單一方法的 GT 不可信。
2. **範圍限定詞寫死**(哪個目錄/檔案),模糊詞(“representative”/
   “one usage”)改成明確集合或「至少 N 個 + 正確分類」。
3. **宣告 ≠ 定義**要在題目與 GT 同時言明;config 相依題必查 build
   系統層(雙閘控)。
4. 每題附 **0-3 評分準則**,把「什麼缺口扣到 2、什麼錯誤打到 0」
   寫進 GT——rubric 不進 GT 就會由評分者即興發揮。
5. **最意外的 GT 發現要由第二人/第二方法複驗**(v3 慣例);GT 修正
   本身也可能錯(WRQ-013 教訓),修正也走同一驗證流程。

## 3. Oracle 也要驗:差分測試法

拿工具當 ground truth 之前,先驗工具:

- **cscope**:對同一 crossref,自寫 parser 與 `-L` 查詢逐符號對拍
  (wpa 全量 ~3,400 查詢)→ 反向抓出 oracle 三類 bug(丟行、雙報
  caller、行號漂移),`cscope-query-engine-bugs.md`,已報 upstream。
- **clangd**:findReferences 實錘回 4/數百且無不完整警告。
- **幻影判定法**(裁決「舊圖 vs 新圖誰對」):對每條消失的邊,檢查
  dst 符號名是否真的出現在該 file:line 的原始碼——機械、可批次、
  不依賴任何工具(v6 前的 D17 驗收:92 個消失站點 100% 判為幻影)。

## 4. 實驗設計協定

1. **隔離**:`git archive` 乾淨樹、`--setting-sources project`(擋
   使用者全域 skill)、工具執行檔路徑不得洩漏原始碼位置線索(cbm
   曾因此跳出沙箱)。
2. **凍結**:prompt 定稿後全量期間一字不改;調整只能發生在 smoke
   階段且逐版記錄動機(adaptation-after-observation 全揭露)。
3. **N=3 取中位數**,報告全距;smoke(N=1)只用於管線驗證,不得
   進結論(v6.5 的 WRQ-019:smoke 3、全量中位 2)。
4. **同場重跑對照臂**:跨場數字只能看趨勢(none 臂 v3→v6 從 54
   漂到 60——模型/環境隨時間變強);要下相對結論必須同場。
5. **成本與時間全記錄**:cost/point 與分數同權重報告;計時用
   `/usr/bin/time -l` + harness 牆鐘雙軌。
6. **resume 機制**以 (題, 臂, rep) 為鍵——長跑必被中斷(session
   limit、機器重啟),可續跑是方法論的一部分。
7. **原始資料全歸檔**:每個 run 的 JSON、每份評分的 JSON、每個
   分析腳本,連同 REPRODUCE.md(前置環境→逐指令→**預期數字**)。
   「可重現」的定義是:別人 clone 後能得到你的數字,而不是相信你。

## 5. 判讀紀律

1. **鑑別力先於總分**:v6 的 22 題有 16 題四臂同分——飽和題庫量到
   的是噪音;報告必須聲明鑑別空間(6 分)並把「換題」列為後續前提。
2. **歸因要到 transcript 層**:分數差要回答「差在哪一步行為」——
   v6 若只看總分會說「LSP 沒用」,看 transcript 才知道是「LSP 沒被
   用」;v6.5 的 WRQ-014 反轉,歸因是 agent 搜索策略方差而非工具。
3. **適用域聲明**:結論 = 條件× 結果。「LSP=grep」只在 *C × 唯讀
   導航 × 中型 repo* 成立,報告若不寫適用域,遲早被拿去錯誤外推
   (外部對照:TS 149K 行上 grep 真的漏引用)。
4. **負結果照登**:60=60、教學層只 +1、自家評分管線踩坑——這些是
   報告可信度的來源;六輪下來最有傳播力的發現全部是負結果。

## 6. 檢查清單(收卷前跑一遍)

- [ ] 全部 runs rc=0 或缺席原因明確(DNF 敘事)
- [ ] 超窗答案稽核(§1.4)
- [ ] 分數/理由矛盾掃描 + 跨 rep 變異 ≥2 逐格複核
- [ ] 反常分數先查 GT(§1.1)
- [ ] 每個對照聲明同場/跨場
- [ ] 適用域聲明寫進 TL;DR
- [ ] 原始資料 + 腳本 + REPRODUCE 歸檔,數字可由腳本重算
- [ ] 修正/勘誤保留舊檔(`*.pre-*`),不覆寫歷史
